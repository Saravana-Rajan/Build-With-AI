Build with AI · Code for Communities · Track 1

Sarvik — Project Blueprint

A multilingual civic-AI platform that turns scattered citizen demands + fragmented public data into a clear, defensible, village-level action plan for a Member of Parliament. The one job: convert subjective political intuition into objective, data-backed constituency planning.

Demo constituency: Dharmapuri, Tamil Nadu Languages: English · Tamil · Hindi Build window: 29 Jun → 8 Jul 2026 Demo Day: 23 Jul · New Delhi Stack: Google Cloud native

The problem, sharply framed

An MP receives development requests through public meetings, letters, social media, grievance portals and direct reps — while the local development plan holds dozens of competing proposed projects. There is no objective way to consolidate feedback, spot recurring needs, and weigh proposals against real demand.

What's broken today

  • Requests arrive through 5+ siloed channels — nothing is consolidated.
  • No way to quantify demand — is "10 asks" 10 people or a whole community?
  • Projects are weighed by gut feel, not data (school vs vocational centre).
  • Citizen asks are never cross-referenced with demographic / infrastructure reality.
  • Money goes to whoever complains loudest; silent, poor, remote areas are ignored.

The insight most teams miss

An MP's own fund (MPLADS ≈ ₹5 crore/year) is tiny next to the hundreds of crores of existing schemes already meant for their constituents — PMAY-G, Jal Jeevan, MGNREGA, pensions.

Most citizen problems are unfilled entitlements, not requests for new money. The failure is delivery & convergence, not budget.

One-line pitch: "We don't help an MP spend ₹5 crore. We help them unlock the ₹500 crore of schemes already meant for their people — and spend their own ₹5 crore only on the gaps nothing else covers."

The outcome we deliver

Route

Send every need to the scheme that should already cover it.

X-Ray

Show, village-by-village, where scheme money is failing to land.

Spend right

Use the ₹5 cr only on true gaps nothing else covers — fairly, silent majority included.

Why it's defensible (a moat, not a feature)

The differentiator is domain insight + hard data integration, not a copyable feature. It understands India's entitlement/scheme system and joins real village-level scheme-coverage data — work most teams won't know to do or can't assemble in time. A policy judge / MP instantly recognises a team that understands how governance actually works. We deliberately avoid fabricated "impact prediction" (e.g. "reduces dropout 12%") — every number we show is computable and real: people served, eligible-but-unserved counts, distance to nearest facility.

The engine — a deliberate two-floor model

We build both floors. Floor 1 is exactly what the problem statement asks for (done solidly → full Problem-Solution-Fit marks). Floor 2 is the differentiator, built on top of Floor 1, never instead of it.

The one-line logic: Floor 1 finds where people are shouting. Floor 2 finds where people are suffering in silence and are legally owed relief.
Floor 1 · what judges expect

Consolidate → Theme → Map → Rank

  • Analyse submissions — Tamil voice/text/photo → structured demand record (Gemini + STT + Translation), auto-categorised.
  • Detect recurring themes — embeddings cluster thousands of submissions by volume + recency. "Drinking water = 400 requests, biggest issue."
  • Map demand hotspots — Google Maps "where the noise is" (Ward 3 = red, 250 road complaints).
  • Combine public data + rank — fuse with Census/NFHS/UDISE + infra gaps + the local dev plan (RAG) → transparent ranked list with "why this rank".
Most teams stop here. Floor 1 alone won't win.
Floor 2 · what wins

Entitlement & equity layer

  • Scheme matching — "whose money is this?" Route each need to PMAY-G / JJM / MGNREGA, or "no scheme → MPLADS".
  • Gap math — eligible − already-covered = entitlements owed but undelivered.
  • Two-track output — Track A (unlock ₹0, push officials) + Track B (spend ₹5 cr only on true gaps).
  • Silent-village detection — high need, zero petitions → equity layer.
  • Anti-lobbying guard — down-weight coordinated submission floods.
  • Close the loop — citizen gets a status update when actioned.
Design rule (non-negotiable): ranking is a transparent weighted formula — Demand × Need × Feasibility + Equity-boost — NOT a black box. Government only deploys what it can explain and audit.

The end-to-end pipeline — input → process → output

LIVE — real time, per submission PRECOMPUTED — nightly batch, cached in BigQuery
Live critical path = Steps 0–3 + Step 10. These must work on stage in real time; Steps 4–9 are cached tables the dashboard just reads. If time slips, cut breadth — never the live intake moment or the X-Ray hero.
Glossary — the terms you'll hit constantly. Click to open.

Track A — unlock money that already exists in a national scheme; cost to the MP = ₹0, the job is to push officials to deliver.

0
Citizen Submission (Intake) Live
+

What & why: Where it all starts. A citizen reports a problem in their own language and gets an instant receipt — we save the raw record first and run the AI in the background, so a slow model never loses someone's submission.

Input
  • Tamil/Hindi/English voice note, text, or photo
  • Metadata: timestamp, channel, lat/lng (if consented)
Process
  • Audio → Cloud Speech-to-Text (ta/hi/en-IN)
  • Translation API → English
  • Write raw record to Firestore immediately, return reference before any AI
  • Fire async task → Steps 1–3
Output
  • Firestore submissions/{id} raw record
  • Citizen sees: "Received. Reference #1042"

Demo shortcut: pre-seed Firestore with the 200–500 synthesised petitions; live submissions are the delta on top. Acceptance: Tamil voice via Telegram → reference < 3s → record visible with English translation.

1
Structured Extraction (Gemini) Live
+

What & why: Turns one messy sentence ("no water near the temple for 3 months") into clean fields — the moment free text becomes data the pipeline can count, sort and route. Gemini is forced to return strict JSON.

Input
  • Raw English text
  • Photo description (Gemini multimodal, if image)
Process
  • Gemini forced-JSON extraction prompt
  • Top-8 category + urgency + beneficiary + location clues
  • is_complaint flag filters spam/test
Output (JSON)
  • category, need_detail, urgency(+reason)
  • beneficiary_estimate, location_clues[]
{ "need_category":"water", "need_detail":"No piped water for 3 months",
  "urgency":"critical", "urgency_reason":"mentions children and summer",
  "beneficiary_estimate":200, "location_clues":["Vellampatti","near the temple"] }

Acceptance: 20 hand-written complaints (3 languages) → ≥18 correct category. Build a small eval set.

2
Geographic Resolution (LGD matching) Live
+

What & why: Pins each complaint to an exact village via its LGD code — the hinge of the whole system. Once tagged, a complaint joins to every public dataset (Census, scheme coverage, schools). GPS first, fuzzy name-match next; never discarded if unclear.

Input
  • location_clues from Step 1
  • lat/lng (if available)
  • LGD master in BigQuery
Process
  • lat/lng → reverse geocode → point-in-polygon (BigQuery GIS)
  • Else fuzzy-match (SOUNDEX/trigram + alias table) Tamil + transliterated names
  • No signal → location_unresolved, bucket to constituency
Output
  • lgd_village_code, village, block, district
  • location_confidence (0–1)
  • Written to BigQuery demand_records

Demo shortcut: scope LGD + GeoJSON to Dharmapuri district only; hand-curate aliases for the ~10 demo villages so they always resolve. Acceptance: 5 demo villages resolve > 0.8.

3
Scheme Matching — "Whose money is this?" Live
+

What & why: The question that makes this project different — whose budget pays? Decides Track A (entitlement exists → push officials, ₹0 of the MP's money) vs Track B (a real gap → the MP's own ₹5 cr). Last step of the live, on-stage path.

Input
  • need_category + lgd_village_code
  • Static scheme routing table
  • Scheme coverage lookup
Process
  • Rule match: category → scheme
  • Gemini eligibility check (Yes/No/Partial)
  • Covered-but-failing → delivery_failure; not covered → coverage_gap; no scheme → mplads_candidate
Output
  • matched_scheme, match_type, track (A/B)
  • Completes the live path → shows on dashboard with scheme tag

Routing: water→JJM · housing→PMAY-G · jobs→MGNREGA · sanitation→SBM-G · health→NHM · education→Samagra Shiksha · pension→NSAP · (none)→MPLADS.

4
Demand Aggregation & Theme Detection Precomputed
+

What & why: Groups thousands of complaints into themes — "water = 400 requests" instead of 400 tickets. Strips duplicates and down-weights coordinated floods so a well-organised lobby can't drown out a genuine majority.

Input
  • All demand_records
  • Run nightly / on-demand button
Process
  • Embed need_detail (Vertex text-embeddings)
  • Cluster (DBSCAN/k-means) → theme per village
  • Recency weight (last 7d × 2)
  • Dedup same submitter+issue; anti-lobbying: >80% similar from different users in 24h → down-weight 0.3×
Output
  • BigQuery theme_clusters: label, volume, recency, coordinated_flag

Acceptance: "drinking water" emerges as top theme by volume — matches the demo narrative.

5
Gap Math (Entitlement Engine) Precomputed
+

What & why: The heart of Floor 2: eligible − already-covered = owed but undelivered, priced in rupees. Produces the headline that makes officials sit up — "₹148 Cr owed but not delivered" — every figure real and auditable, no predictions.

Input
  • Census (eligible population)
  • Scheme MIS (covered)
  • Per-unit values (auditable table)
Process (per scheme per village)
  • gap = eligible − covered
  • gap_value = gap × per_unit_value (PMAY-G ≈ ₹1.2L/house)
  • BigQuery SQL join on lgd_village_code
Output
  • BigQuery scheme_gaps
  • Headline = SUM(gap_value) → "₹148 Cr owed but undelivered"

Acceptance: constituency headline ₹ renders; Vellampatti shows a large defensible PMAY-G/JJM gap.

6
Silent Village Detection Precomputed
+

What & why: The equity feature, and the demo's emotional peak. Normal systems reward whoever complains loudest; this one inverts that — it rewards need and penalises complaint volume, so villages with high need but zero petitions (too poor or remote to ask) rise to the top instead of disappearing.

Input
  • Petition counts per village
  • scheme_gaps + Census deprivation
Process
  • silent_score = need_score × (1/log(petition_count+1))
  • petition_count = 0 AND need_score > threshold → silent_flag
Output
  • BigQuery silent_villages
  • Equity hero feature

Demo shortcut: ensure Vellampatti = 0 petitions, need_score ~0.91, silent_flag TRUE — the mic-drop moment.

7
Priority Scoring & Ranking (+ RAG) Precomputed
+

What & why: Pulls everything into one ranked to-do list, blending Demand + Need + Feasibility + Equity into a single transparent, tunable score. RAG checks the official dev-plan so we link to real projects, not invented ones — and every rank can show its "why" (government deploys only what it can explain).

Input
  • theme_clusters + scheme_gaps + silent flags
  • Dev-plan PDF indexed in Vertex AI Search
Process — transparent formula
  • Priority = (W1·Demand + W2·Need + W3·Feasibility + W4·Equity) × Scheme_Deduction
  • W1=.30 · W2=.35 · W3=.20 · W4=.15 (tunable)
  • RAG: map to existing proposed project — don't invent
Output
  • BigQuery ranked_projects with why_json factor breakdown for the UI

Acceptance: top-4 list matches narrative; each row expands to factor contributions; weights live in config (show "what if equity mattered more?").

8
Budget Optimiser — Track B Precomputed
+

What & why: For Track B only (the MP's ₹5 cr). Picks the set of projects that maximises total priority without exceeding the ₹5 cr cap — a classic "knapsack", solved with OR-Tools — giving a ready-to-present spending plan.

Input
  • ranked_projects WHERE track='B'
  • ₹5 cr cap · PWD/CPWD cost estimates
Process
  • OR-Tools knapsack: maximise Σ priority s.t. Σ cost ≤ ₹5 cr
Output
  • mplads_portfolio: selected projects, cost+score each, total used, contingency
Road Vellampatti–Pennagaram ₹1.8 Cr (74) · School Karimangalam ₹1.2 Cr (68)
Community hall Palacode ₹0.9 Cr (61) · Drainage Harur ₹0.8 Cr (58)
Total 4.7 / 5.0 Cr · Remaining ₹0.3 Cr contingency

Honest risks & rebuttals

"Idea is graspable once spoken"

Protect with execution depth (real data). Don't over-explain the concept publicly before submission.

"This is just convergence / DPC planning"

"Yes — and it fails because nobody joins the data at village level. We did."

"Hardest work = messy data joins"

Tamil → LGD codes → scheme coverage. Front-load it; budget time here, not on UI.

The real goal

"Unique" is not the bar — "an MP would use this Monday morning" is.
